## Overview

Stingle Photos is an end-to-end encrypted photo/video backup app for iOS. Media is encrypted client-side with libsodium before it ever leaves the device; the server only stores ciphertext. Most app logic lives in a shared framework (`StingleRoot`) consumed by both the main app and the share extension.

## Build, Run & Test

There is no Podfile or `Package.swift`; dependencies are resolved via Xcode-managed Swift Package Manager, pinned to released versions (Alamofire 5.12, keychain-swift 24, swift-sodium 0.11 — previously tracked `master`). Open `Stingle.xcodeproj` directly — there is no `.xcworkspace`.

Targets build with `IPHONEOS_DEPLOYMENT_TARGET = 16.0`, `SWIFT_VERSION = 5.10`, and `SWIFT_STRICT_CONCURRENCY = targeted` (clean). It builds against the current iOS SDK, so behavior follows the latest UIKit/AVFoundation even though the floor is iOS 16.

This repo requires the full Xcode toolchain. `xcodebuild` from the command line needs Xcode selected first (`sudo xcode-select -s /Applications/Xcode.app/Contents/Developer`); the machine may default to CommandLineTools, which cannot build it.

### Deploying to verify (important gotcha)
A command-line `xcodebuild ... -derivedDataPath build` only proves the code **compiles** — it writes to `./build` and does **not** update what the user runs from Xcode. The user installs via **Xcode ⌘R**, which uses Xcode's *own* DerivedData. So "my build succeeded" ≠ "the app on the device changed." If reported changes don't appear (a recurring trap here): the build is stale — **Quit Xcode fully (⌘Q), reopen, Clean Build Folder (⇧⌘K), ⌘R.** Also confirm the running app's logs reflect your edits before chasing a "bug" — it's often just a stale build.

To run on a simulator yourself for verification:
```
xcodebuild -project Stingle.xcodeproj -scheme StingleDev -configuration Debug-Dev -destination 'platform=iOS Simulator,name=iPhone 17' -derivedDataPath build build
xcrun simctl install "iPhone 17" build/Build/Products/Debug-Dev-iphonesimulator/Stingle.app
xcrun simctl launch "iPhone 17" org.stingle.photos.dev   # dev bundle id
xcrun simctl io "iPhone 17" screenshot /tmp/shot.png
```
The simulator typically has a saved locked account (`test1@fenritz.com`); real login/sync/upload need credentials to runtime-verify, and the gallery's StoreKit plans need either App Store Connect or the local `Stingle/Resources/Config/Storage.storekit` (already wired into the `StingleDev` scheme).

Targets and schemes:
- **Stingle** (`com.apple.product-type.application`) — the main app. Schemes: `StingleDev`, `StingleProd`.
- **StingleShare** (`app-extension`) — share-sheet extension for importing media. Schemes: `StingleShareDev`, `StingleShareProd`.
- **StingleRoot** (`framework`) — shared core (crypto, DB, network, managers, models, data sources). Scheme: `StingleRoot`. Most non-UI code lives here.

Build configurations are split by environment: `Debug-Dev`, `Debug-Prod`, `Release-Dev`, `Release-Prod`. Dev vs. Prod is driven by xcconfig files in `Stingle/Resources/Config/` (`Development.xcconfig`, `Production.xcconfig`), which set bundle IDs, app name, API URL, and `BUILD_TYPE`. These values are read at runtime through `STEnvironment` (via Info.plist keys like `BUILD_TYPE`, `BASE_API_URL`, `APP_BUNDLE_ID`). Use the `*Dev` schemes for development against the dev backend.

There is no test target in this project; do not assume a `test` action exists.

## Architecture

### App / extension / framework split
- The main app and share extension share an **app group** (`group.<bundleId>.sharing`, see `STEnvironment.groupAppFileSharingBundleId`) for the encrypted file store, UserDefaults, and keychain. Both targets link `StingleRoot`.
- `STEnvironment.current.appIsExtension` distinguishes runtime context (detected by `.appex` bundle suffix). Be careful with code paths that differ between app and extension.

### STApplication — the composition root
`STApplication.shared` (in `StingleRoot/.../Services/Application/`) is the singleton that wires everything together and owns the long-lived services: `crypto`, `dataBase`, `syncManager`, `uploader`, `downloaderManager`, `autoImporter`, `appLockUnlocker`, and a per-user `fileSystem`. Logout/delete-account flows that must tear down all state live here (`logoutCashe`/`deleteAccountCashe`). When adding a new global service, register it here.

### Crypto (`Uitils/Crypto/STCrypto*.swift`)
libsodium-based (swift-sodium / Clibsodium). Implements Stingle's custom encrypted file format — magic bytes `SP` for media files and `SPK` for key files, chunked XChaCha20-Poly1305 (`__data__` context), versioned headers, and KDF difficulty levels. Keys derive from the user's password + a mnemonic backup phrase. `STCrypto` is split across extensions by concern: `+Header`, `+PrivateKey`, `+Album`, `+Common`, `+Additional`. Format constants (lengths, versions, file-type tags) live in `STCrypto.Constants`. Do not change format constants without understanding on-disk/wire compatibility.

### Persistence (`Manager/DB/`, `Models/DB/`)
Core Data, model name `StingleModel` (`.xcdatamodeld` in `StingleRoot/Resources/DBModel/`, with a migration mapping model present). `STDataBase` exposes typed **providers** — `userProvider`, `dbInfoProvider`, and `SyncProvider`s for gallery, albums, albumFiles, trash, and contacts. Each library entity has a Core Data class (`STCD*`) and a plain model (`STLibrary.*`) bridged via `CDConvertable`/`ICDSynchConvertable`. Sync diffs are modeled as `SyncInfo` sets (inserts/updates/upgrade/deletes). There is **one** `NSPersistentContainer` with a single `viewContext` (main, FRC-backed gallery) and a single shared `backgroundContext` (all sync writes); both hit the same store, so a heavy read/write on one can lock-contend with the other on the main thread.

**Threading gotchas (each caused a real multi-second UI freeze):** (1) `STDataBase.sync` persists with `context.perform { try context.save() }` on the **background** queue — it must NOT hop to main and `performAndWait` the save (the old code did, blocking main for the whole save; a first/large sync froze the UI for seconds). Only the post-save UI work (`endSync` → the gallery FRC `performFetch`/diffable apply, which *does* have to be on main) hops back via `DispatchQueue.main.async`. (2) `CollectionProvider.fetchObjects(...)` builds the plain models **inside** the context's `performAndWait` — `Model(model:)` reads managed-object properties and is only safe on the owning queue; doing it after the fetch returned was a threading violation that also contended with the main-thread merge. It takes an optional `batchSize` (faults rows in chunks) and now actually honors the passed `context`. For read-only full-catalog scans use `STDataBase.makeReadContext()` (a fresh throwaway context that doesn't auto-merge) so the scan never serializes behind the shared `backgroundContext`. (3) The gallery FRC (`STDataBase.DataSource.createResultsController`) sets `fetchBatchSize` — `finishSync` calls `controller.reloadData()` → `performFetch()` on the **main** thread on *every* sync, and without a batch size that materialized every object's property values at once (the remaining multi-second freeze after the save fix). Batching faults rows in as cells scroll in; the diffable snapshot only needs object IDs. Note `performFetch` is **required** here (don't delete it): the catalog is written with `NSBatchInsertRequest`/`NSBatchDeleteRequest`, which bypass the contexts, so the `viewContext` only sees new/removed rows after a re-fetch — `automaticallyMergesChangesFromParent` does **not** surface batch-op changes (the `backgroundContext` is a *sibling* of `viewContext` on the PSC, not its parent; there is no context-merge anywhere). (4) **`finishSync` only re-fetches providers that actually changed.** `endSync(info:)` passes each provider's `SyncInfo.hasChanges` to `finishSync(reload:)`, which skips `performFetch` when that provider's Core Data was untouched. Without this, *every* sync — including the no-op delta that an app-open sync usually is — did a full-catalog `performFetch` on the main thread for nothing (the recurring "small freeze"). It's safe because `SyncInfo` is empty **iff** no batch insert/delete/`object.update` ran (the `willFinish` file moves/deletes are filesystem-only and derived from the same sets), and every other DB mutation (uploads, user edits, `restoreFilesIfNeeded`) drives its own `reloadData`. Relatedly, `STFileUploader.uploadAllLocalFiles()` (called from `didEndSync` on main every sync) runs its `canUploadFile()` guard + `getLocalFiles()` fetch on its background queue, not main. (5) **`DBInfoProvider.update(model:)` persists on the `backgroundContext`, never the main `viewContext`.** It's called once per uploaded file (from the uploader's queue, for the `spaceUsed` the server returns on each upload). The old code ran a synchronous `performAndWait` on the main `viewContext` and then `context.reset()` — and `reset()` tears down every managed object the gallery FRC/visible cells hold, forcing a full refault. In the completion burst at the end of a large upload that ran dozens of times back-to-back on main and froze the UI ("freeze when upload finishes"). The in-memory `myDBInfo` is updated synchronously so readers/observers see the new value immediately; the background save auto-merges the single `STDBInfo` row into the viewContext. Don't route the at-rest dbInfo write back through `viewContext`, and never `reset()` the viewContext to "clean up" a one-row write. (6) **The gallery/trash/albumFiles FRC `sectionNameKeyPath` (`STCDFile.day`) is a *computed* property, not a stored attribute — this was the dominant root cause of the upload/sync freezes.** All three CD classes inherit `STCDFile`; `day` formats `dateCreated` ("MMM dd, yyyy"). Because it isn't in the SQL store, `performFetch()` (which runs on the **main** viewContext on every reload — incremental sync, *and* the per-batch reload at the end of an upload) cannot section in SQLite: it must evaluate `day` for **every** object, faulting the whole library and running a `DateFormatter` per row on main. `fetchBatchSize` does **not** help — section evaluation forces full faulting regardless. It was made pathological by `STDateManager.dateToString` reassigning `dateFormat` on every call (re-parsing the ICU pattern × N). Fix (in `STGalleryVM.swift`'s `extension STCDFile`, **keep the section strings identical**): a dedicated `DateFormatter` whose `dateFormat` is set **once**, plus an `objectID`-keyed `NSCache` memo. A cache hit returns from `objectID` **alone** (never touches `dateCreated`, so it does **not** fault the row) — valid because `dateCreated` is immutable per file. Once the gallery's first load warms the cache, later `performFetch`es re-section existing rows from cache without faulting; only genuinely new rows (bounded by the upload batch, not library size) miss. If you ever change `day`'s output you must invalidate the cache. The proper long-term fix is to promote `day` to a stored attribute populated at insert (needs a schema migration + backfill of every batch-insert path — deferred). (7) **The uploader batches its Core Data writes so the gallery FRC fires per *batch*, not per *file*.** Correction to (3)'s aside: a **normal** `context.save()` on the `backgroundContext` *does* auto-merge into the `viewContext` (`automaticallyMergesChangesFromParent = true`) and *does* fire the gallery FRC — it's only **batch** insert/delete that bypass it. So `STFileUploader.updateDB`, which called `galleryProvider.update(models:[singleFile])` once per uploaded file, triggered a backgroundContext save → viewContext auto-merge → full-snapshot FRC reload **on every file** (an O(library) main-thread reload per file — the "freezes on every file" report; the explicit `reloadData` flag was already throttled to every 5, so the per-file cost was purely the auto-merge). Fix: `updateDB` now accumulates successfully-uploaded files into `pendingDBUpdateFiles` (per `DBSet`) and flushes them through a **single** `update(models:)` every `maxCountUpdateDB`, with a final `flushAllPendingDBUpdates()` when `uploadingFiles` drains (called from **both** the success and failure handlers so a trailing partial batch — or one stranded by a failed last file — is never lost). One save per batch ⇒ one auto-merge ⇒ one FRC reload per batch. Persistence of `isRemote/isSynched` lags by < a batch; a mid-upload kill self-heals (next launch's sync reconciles already-uploaded files before `uploadAllLocalFiles` re-sends them). This composes with (6): each batch reload is also faultless thanks to the `day` cache. Don't revert to per-file `provider.update(...)` in the uploader. **Update (item-level rewrite):** the gallery FRC no longer does a *full-snapshot reload* on a normal `save()` auto-merge — the item-level delegate now emits the uploaded files' `isSynched`/`isRemote` flip as **updates**, which `STCollectionViewDataSource` applies by reconfiguring just the visible cells (no `performFetch`, no relayout — see "Gallery data source: item-level" under UI). So the per-file cost the uploader batching was fighting is now near-zero regardless; batching still helps (fewer `backgroundContext` saves/auto-merges), and the `day`-cache in (6) now only matters on the remaining full-reload paths (initial load / unlock / sync / a too-large delta), not per upload.

### Sync
`STSyncManager` reconciles local Core Data with the server. The server is authoritative for the encrypted catalog; the client pulls deltas keyed off `STDBInfo` timestamps and applies them through the sync providers. Sync runs on unlock and is gated by `appLockUnlocker`.

### Network (`Network/`)
Layered: `Request/` (typed `IRequest` definitions per domain — Auth, Album, File, Sync, Upload, etc.) → `Dispatcher/` (`STNetworkDispatcher` + `STNetworkSession`, Alamofire-backed, with request encoding) → `Worker/` (high-level domain operations like `STSyncWorker`, `STUploadWorker`, `STContactWorker`) → `Response/` (`IResponse`/`STResponse`). `Operations/` (`STOperationManager` + queues) drives concurrent background work for uploads/downloads.

### Concurrency / async-await (cross-cutting)
The codebase was historically all completion-handler callbacks. The worker layer now also exposes `async throws` siblings (in `STWorker` and each worker) that **bridge the existing callback methods via `withCheckedThrowingContinuation`** — both styles coexist so migration is incremental. View models call the async API inside `Task { @MainActor [weak self] in ... }`, which preserves the original "callbacks fire on main" semantics (the operation layer doesn't hop threads; Alamofire's default queue is `.main`).

**Continuation hazard:** a bridged callback method *must* invoke success XOR failure exactly once. A path that fires both ⇒ `continuation.resume` twice ⇒ **crash**; a path that fires neither ⇒ leaked continuation/hang. The operation layer nils its closures in `setFinished` (so post-success cancel is safe), and `STOperation.setCancel` fires a `.cancelled` failure on ready-cancel so awaiters don't leak. When you bridge or edit a callback worker method, verify every branch resolves exactly once.

### File transfer & storage
- `Manager/FileUploader/` — `STFileUploader` + per-file `STFileUploaderOperation`; encrypts and uploads.
  - **Uploading is gated separately from auto-import.** `canUploadFile()` requires **backup** enabled (`STAppSettings.backup.isEnabled`, *not* the auto-import toggle), Wi-Fi if Wi-Fi-only, and battery ≥ threshold. The battery check uses `UIDevice.current.batteryLevel`, which is **-1 unless `isBatteryMonitoringEnabled`** — enabled in `AppDelegate` at launch, and `canUploadFile()` ignores an unknown (-1) level so a cold background launch isn't blocked.
  - **Background uploads.** File upload requests are always `isBackground = true` and run on the `URLSessionConfiguration.background` session in `STNetworkSession` (`sessionSendsLaunchEvents = true`, app-group `sharedContainerIdentifier`), so byte transfers continue after suspension. `AppDelegate.application(_:handleEventsForBackgroundURLSession:…)` forwards the system completion handler via `STNetworkDispatcher.handleEventsForBackgroundURLSession(...)` so completions get finalized when iOS wakes the app — without this, transfers finish but the local→remote DB finalization waits for the next foreground launch. Note the BG auto-import task signals completion when *encryption/import* finishes, not when uploads finish; uploads then ride the background session independently.
- `Manager/FileDownloader/` — `STDownloaderManager` with a `Cache/` layer for decrypted thumbnails/originals.
  - **Background thumbnail sync.** `STThumbnailSyncManager` (StingleRoot, registered on `STApplication.shared.thumbnailSyncManager`) walks the whole catalog (gallery + albumFiles + trash via `*Provider.fetchObjects()`) and downloads the encrypted thumbnail of every remote+synced file missing from the on-disk cache (`fileSystem.isExistFile(file:isThumb:true)`). It owns its **own dedicated `STDownloaderManager.FileDownloader` instance** so its progress is isolated from user-initiated original-file downloads. Triggered from `STSyncManager.didEndSync` (every sync attempt) and cleared on logout. Like the camera path it only needs the *public* key, so it works while app-locked. `startSync()` is idempotent: one batch in flight at a time; mid-batch re-triggers coalesce into a single rescan on completion (`needsRescan`) to avoid re-enqueuing in-flight thumbs. **Critical gotchas (each caused a real UI freeze / bug):** (1) it feeds the downloader through a **bounded in-flight window** (`windowSize`, refilled from a `pending`/`pendingIndex` cursor on each terminal callback) — do **not** hand the whole missing-thumbnail list to `FileDownloader.download(sources:)` at once, since on a large library that runs a synchronous per-source enqueue loop and floods the main queue with per-source callbacks, hanging the UI. The downloader's `didFinished` (fires only when the window has drained `pending`) is the authoritative end-of-batch signal. (2) The scan must stay **off the main thread and off the shared context.** It reads through `STDataBase.makeReadContext()` (a dedicated throwaway context), filters to remote+synced rows in SQLite (`isRemote == YES AND isSynched == YES` predicate, not a fetch-all-then-filter-in-Swift), and faults in batches (`fetchBatchSize`). An earlier version fetched the whole catalog and built the plain models *off* the context queue — a Core Data threading violation that lock-contended with the main-thread gallery merge and froze the UI for seconds ~2s after sync (and made thumbnails "start very late"). It's still **deferred (`scanStartDelay`, 1.0s) and runs at `.utility` QoS** so the first download burst doesn't pile onto the gallery's post-sync reload, but the delay now only has to outlast that reload, not a main-blocking scan. (3) **Cache-poisoning guard:** a rate-limited server can answer a thumbnail request with a JSON error body at HTTP 200, which Alamofire writes straight to the thumb cache path (Alamofire downloads aren't `.validate()`d here). Every finished download is checked against the Stingle magic (`STCrypto.Constants.FileBeggining` = "SP"); a non-matching file is deleted (so the next scan retries) and the rest of the batch is aborted to stop hammering the server. Progress is surfaced via `IThumbnailSyncObserver` and shown as a "Thumbnail sync" row (label + progress bar, no thumbnail) at the top of the sync-status dropdown (`STUploadsVC`/`STUploadsVM`, reusing `STUploadsTableViewCell` — the cell's `CellModel.showsImage` flag hides the grey `STPlaceholder` image container for status rows).
- `Manager/FileSystem/STFileSystem` — per-user on-disk layout under the user's `homeFolder`; handles migration, logout cleanup, free-disk checks (`STConstants.minFreeDiskUnits`).
- `Importer/` — `STImporter` and `STFileAutoImporter` (`AutoImporter`) ingest from the photo library / share extension, including background auto-import.
  - **Auto-import triggering (reliability gotchas, learned the hard way).** `AutoImporter.startImport()` is the single entry point and self-guards via `canStartImport` (logged-in + auto-import enabled + filesystem available). It's now driven by four triggers so it doesn't stall when the app is closed or idle: (1) a `PHPhotoLibraryChangeObserver` on `AutoImporter` (so `AutoImporter` is an `NSObject`) imports new media live; (2) `STSyncManager.didEndSync` calls it after *every* sync attempt — success **and** failure (don't move it back into the sync success closure, or a failed/offline sync skips import); (3) two `BGTaskScheduler` tasks in `STBGTaskScheduler` — a `BGProcessingTaskRequest` (`org.stingle.photos.auto.import`) and a `BGAppRefreshTaskRequest` (`…import.refresh`); both ids must stay in the app `Info.plist` `BGTaskSchedulerPermittedIdentifiers` and `UIBackgroundModes` must include both `processing` and `fetch`. Background processing no longer sets `requiresExternalPower` (it used to, so imports only ran while charging). **`STBGTaskScheduler.shared.start()` (which calls `BGTaskScheduler.register`) must run synchronously inside `AppDelegate.didFinishLaunchingWithOptions` — it was previously wrapped in `DispatchQueue.main.async`, which registers the handlers one run-loop tick too late, and iOS then silently never launches the app to run them (the #1 reason "background auto-import doesn't work at all").** Note iOS also won't run BG tasks after the user force-quits the app, and BGTaskScheduler never fires on the Simulator (test on a real device, or trigger via the LLDB `_simulateLaunchForTaskWithIdentifier:` private call) — foreground/observer triggers are the real safety net. Background import doesn't need an app unlock: `canStartImport` is Core-Data-backed and new files seal to the user's public key. (4) the settings toggle.
  - `startImport()` coalesces re-triggers during an in-flight run via `needsRestart` (re-runs once on completion) so a change/sync arriving mid-import isn't dropped.
  - **Detection is creationDate-paged + identifier-deduped.** The scan fetches `creationDate >= lastImportDate` ascending and skips assets whose `localIdentifier` is in a persisted, bounded (`importedIdentifiersCap`) set (`UserDefaults` key `autoImporter.importedIdentifiers`). The `>=` (was `>`) plus dedup fixes assets that share the boundary timestamp being silently skipped, without re-importing or looping. `resetImportDate` clears the id set (logout / "reset import date" / enabling auto-import, which baselines `lastImportDate` to *now*). Known limitation: media saved *now* but with a `creationDate` older than the baseline (Messages/AirDrop/iCloud) is intentionally not imported — "import from enable point" semantics.

### App lock
`Manager/AppLockUnlocker/` + `STBiometricAuthServices` implement passcode/biometric locking. Locking pauses sync and clears in-memory key material; unlocking re-triggers sync and DB reload (`STApplication.appDidLocked`). The biometric credential is an encrypted password in app-group UserDefaults with its key in the keychain (`STKeyChainService`). Three non-obvious rules learned here: (1) that keychain item must use `accessibleAfterFirstUnlockThisDeviceOnly` (not the `WhenUnlocked` default) or Face ID fails after a long device lock and falls back to the password dialog; (2) decide whether to *offer* biometrics from `isBiometricConfigured` (UserDefaults-backed, reliable) rather than a read-and-validate that can fail transiently, and defer the auto-attempt until `protectedDataDidBecomeAvailable`.

(3) **The auto-lock timer arms on `didEnterBackgroundNotification` (actual background), NOT `willResignActiveNotification` — and the timestamp is consumed once resolved.** `STAppLockUnlocker.Locker` measures background time via a single `backgroundDate`, set when the app backgrounds and cleared (in `evaluateAutoLockIfNeeded`) the moment a foreground transition is resolved, whether or not it locked. This fixed a "returning from the Camera tab sometimes triggers Face ID" bug. Root cause was an **asymmetric notification pair**: the timer armed on `willResignActive` (which fires for *transient* interruptions that never background the app — Control Center, the notification shade, banners, and the first-time camera/mic **permission dialog**) but only resolved on `willEnterForeground` (fires only on real background return). So a transient interruption planted a `resignActiveDate` that nothing ever cleared. The camera tab pauses auto-lock (`STCameraVC` toggles `isAutoLockPaused`); **leaving the tab runs the one path that consults the timestamp outside normal background→foreground gating** (`isAutoLockPaused` `didSet` → `evaluateAutoLockIfNeeded(ignorePause: true)`). With the app continuously *foreground* the whole time, that measured stale-resign-to-now wall-clock (i.e. foreground time on the camera), and if it exceeded the auto-lock interval (always, for "immediately" = 0) it locked → Face ID. Arming on `didEnterBackground` makes the pair symmetric with `willEnterForeground` so transient interruptions never arm the clock; consuming `backgroundDate` on resolve stops a stale timestamp from locking a later foreground-only transition. The legitimate case still holds: a *real* background while on the camera arms `backgroundDate`, the pause defers the decision, and leaving the tab locks if the background exceeded the timeout (watchdog-capped at 5 min). Don't revert the arm trigger to `willResignActive` — the privacy-cover snapshot (SceneDelegate) is what should key off resign-active, not the lock clock.

### Video playback (`Uitils/Player/`)
`STPlayer` wraps `AVPlayer`; `STAssetResourceLoader` is an `AVAssetResourceLoaderDelegate` that decrypts the chunked stream on the fly (local file or network range-request reader). Each chunk decrypts independently (per-chunk key derived from the chunk number), so the asset **does** support random access — `contentInformationRequest.isByteRangeAccessSupported` must be `true` for scrubbing to work at all. Seeking is coalesced ("chase time") with a tolerance so a drag doesn't flood `AVPlayer`. Video viewers (`Items/Video/STVideoViewerVC`) autoplay when they become the current page (`fileViewer(activateContent:)` from `STFileViewerVC.didChangeFileViewer`) and pause off-screen.

**NetworkReader chunk reassembly (the "some videos load forever from the server" bug).** `STNetworkDataTask` streams a range response incrementally (`URLSession didReceive` fires per packet), and `NetworkReader.didReceiveNewData` reassembles whole encrypted chunks before handing each to the `Decrypter` (which needs a chunk whole and in order, since the per-chunk key is derived from its sequence number). The original emit test was `receiveData.count / chunkSize == 1` — it only flushed when *exactly one* full chunk was buffered. A single network delivery that buffered **two or more** chunks at once (bursty CDN/TCP, or post-background catch-up) skipped past `== 1`, emitted nothing, and the buffer only grew → the resource-loading request was never fulfilled and `AVPlayer` waited forever (surfacing as `PlayerRemoteXPC`/`Fig` errors, not a crash). Fix: drain **all** complete chunks in a `while` loop, flush the short final chunk at end-of-data, and call the handler **synchronously** on the reader's queue (it already runs under a `.barrier`), which also removed a latent out-of-order hazard from dispatching each handler async onto a concurrent queue. Requested ranges are always chunk-aligned (`length = nChunks * chunkSize`), so mid-stream the buffer drains to empty and only the true final chunk is ever partial.

**Played remote videos are cached.** A network-streamed video (`STAssetResourceLoader.NetworkReader`) decrypts chunks on the fly and **discards** the encrypted bytes — nothing persists, so re-watching re-streams from scratch. `STPlayer.cacheCurrentFileIfNeeded()` (guarded once-per-file by `didStartCaching`) fixes this: when a remote file isn't already on disk (`!assetResourceLoader.isLocalPlaying`) it enqueues a one-time full-file download via `downloaderManager.videoCacheDownloader`, which writes the encrypted original to `…/server/oreginals` (the same path `LocaleReader` reads from). The download rides that shared downloader (observers are weak), so it completes even if the viewer/player is torn down mid-watch. Only `isRemote && isSynched && not-yet-cached` files trigger it (local/unsynced files already exist on disk → skipped). It does **not** gate on Wi-Fi-only — caching a watched video can use cellular.

**Caching is on a dedicated single-concurrency downloader (`videoCacheDownloader`), not the 20-wide `fileDownloader` (pile-up gotcha).** Each watched video enqueues a *whole-file* download; on the shared `fileDownloader` (`maxConcurrentDownloads` 20) watching many videos in a row ran up to 20 full downloads at once, saturating bandwidth and making each *new* video's stream start progressively slower (observed time-to-first-frame climbing 7s→14s across a session). `STDownloaderManager.videoCacheDownloader` is a `VideoCacheDownloader: FileDownloader` overriding `maxConcurrentDownloads` to **1**, so caches process one at a time and the foreground stream competes with at most one background cache download. `maxConcurrentDownloads` is an overridable property on the base `Downloader` (default 20; don't lower it on `fileDownloader` — that would serialize user-initiated multi-file downloads). The player observes `videoCacheDownloader` (not `fileDownloader`) for the swap.

**Caching is deferred past a "settle" delay, NOT started at `play()` (bandwidth + skim gotcha).** Starting the whole-file download the instant `play()` was called made it race the live stream's range requests for first-frame bandwidth. Now `play()` only arms `scheduleCacheDownloadFallback()` (a `cacheStartFallbackDelay` = 10s backstop that caches anyway if playback never began — `state == .playing && !didStartCaching` — so the rescue swap can recover a stalled stream). On first frame (`timeControlStatus == .playing`), `scheduleCacheDownloadAfterSettle()` waits `cacheStartSettleDelay` (3s) and only then caches *if still playing the same file* — so skimming quickly through a feed of videos never enqueues them, and a real watch caches without stealing startup bandwidth. (The earlier "start at first frame" approach cached every glanced-at video.)

The auto-cache download is **silent**: `FileDownloader.download(files:showsProgress:)`/`download(sources:showsProgress:)` take `showsProgress` (default `true`); the player passes `false`. A silent download still fetches+caches the file and still fires the terminal `didEndDownload` to observers (so the player swaps to local), but it stays out of the progress accounting entirely — no `progressesCount`/`progresses`, no `didStartDownload`/`didChangeProgress`/`didFinished`. (The dedicated `videoCacheDownloader` isn't observed by the gallery nav-bar bar — `STGalleryNavigationController` watches `fileDownloader` — so caching can't surface progress regardless; `showsProgress:false` is kept for belt-and-suspenders.) Only user-initiated downloads (Download button, album/share export) show the bar. The silent terminal notify takes a dedicated path (`notifyEndDownloadWithoutProgress`) that skips the `progressesCount -= 1` math (it was never incremented). Caching is gated by **`STAppSettings.advanced.autoCacheVideos`** (Advanced settings, default on; switch row in `STAdvancedVC`). Because `Advanced` is `Codable`-persisted as a dict and old installs lack the key, `Advanced.init(from:)` is custom: it `decodeIfPresent`s the new flag (default `true`) so a missing key doesn't fail the whole decode and silently reset `cacheSize`.

`STPlayer` is *already* a `STFileDownloaderObserver` whose `downloader(didEndDownload:)` can swap the live item from the network stream to the now-local file. **The swap is gated on "playback hasn't begun yet"** (`state == .playing && !hasStartedPlayback`, where `hasStartedPlayback` flips true the first time `timeControlStatus == .playing` and resets per file in `replaceCurrentFile`). Two reasons: (1) the cache download usually *wins the race* against the AVPlayer network item becoming ready — and that item sometimes never readies (the "video won't start until I pause+play" bug), so swapping to the reliable local file and calling `play()` both speeds up and un-sticks the first play; (2) swapping mid-playback builds a fresh `AVPlayerItem` at zero, so doing it after frames are on screen would visibly reload and restart from the beginning — so once playback has begun we leave the network stream running to the end and let the cache serve the *next* open. The old handler instead resumed only `if isPlaying`, which both failed to start the common buffering case (download finished before `.playing`) and restarted an already-playing video. A pre-play scrub position (slider dragged while buffering) is preserved best-effort via `seek`.

**Streaming session is shared across videos.** `STAssetResourceLoader.networkSession` returns a single static `sharedStreamingSession`, not a per-loader one (keeps connections warm; URLSession is thread-safe, tasks cancelled individually, session never invalidated — don't revert to per-instance/`lazy`). NOTE: this did **not** move the needle on startup, because the per-video ~1.0–1.6s first-byte latency is **Wasabi cold-object latency**, not client connection setup — the trace proves it: connection reuse already works (the *2nd* range GET of each video is ~0.2s on the warm connection), but each *new object's* first GET is ~1.0–1.6s regardless of session sharing. That cost is server-side and not fixable here. Other inherent, unoptimized startup costs once observed via tracing: the content-information request (AVPlayer asks 2 bytes) forces a full 1 MB chunk download to decrypt+serve them, chunk 0 is then re-downloaded for the first data request, and non-fast-start videos (moov at end) must fetch the tail before they can begin (~10s to first frame for a large one). (A temporary `#if DEBUG` `STStreamDebug` trace through the streaming/decrypt path was used to diagnose all of the above and has since been removed; re-add timestamped logging around `STAssetResourceLoader`/`Decrypter`/`NetworkReader` if you need to re-measure.)

### Gallery thumbnail rendering & the "grey flash" (UI gotcha)
Grid cells load thumbnails through `UIImageView.setImage(source:)` (`UIImageView+Operation.swift`) → `STDownloaderManager.ImageRetryer`. The default path sets `image = nil`, starts a grey placeholder animation, and resolves through the operation queue + `DispatchQueue.main.async` — **even for an in-memory cache hit**. That's invisible for one cell, but when a whole sync reload reconfigures every visible cell at once the entire gallery flashes grey. Two guards keep it from regressing: (1) `setImage` has a **synchronous fast path** — `ImageRetryer.memoryCachedImage(source:)` reads the decoded `NSCache` (`DiskImageCache`, countLimit 400) and, on a hit, paints it immediately with no nil-gap/animation (it also cancels any in-flight op so a late callback can't overwrite it). **It is gated on `success == nil`** — only fire-and-forget display callers (grid cells) take it. Callers that pass a `success` closure drive a load lifecycle and rely on the original async timing: e.g. `STPhotoViewerVC` sets the image in `viewDidLoad`, and a *synchronous* set there runs before the zoom view (`STImageZoomView`/`STZoomView`) has real bounds, which sizes the image to **zero → a black frame** until the full-res arrives and re-lays-out. (2) grid cells (`STGalleryCollectionViewCell`) track `currentImageIdentifier` and skip `setImage` when the image's identifier (`version_folderType_fileName`, so it changes exactly when the thumbnail does) is unchanged — the FRC marks a cell "reloaded" for metadata-only updates (e.g. `isSynched` flipping after upload) and a bare `setImage` would needlessly re-fetch. Building a `CellModel`/`STImageView.Image` decrypts the file headers (libsodium) on the main thread, so keep per-cell reconfigure work minimal.

### Gallery data source: item-level, not diffable-snapshot (the scale-to-millions rewrite)
`STCollectionViewDataSource` is **no longer `NSDiffableDataSource`-backed**. A diffable `apply` (and the old `controller(_:didChangeContentWith:)` snapshot delegate) is **O(total items)** — it diffs the whole library on every change, which froze a 25k+ gallery on each upload and doesn't scale to the multi-Tb (millions-of-items) libraries the product sells. It was rewritten to the FRC's **item-level delegate** (`controllerWillChangeContent`/`didChange anObject`/`didChange sectionInfo`/`controllerDidChangeContent`), which reports **O(changes)**. A small non-generic `STCollectionViewDataSourceAdapter` is the actual `UICollectionViewDataSource` (a generic class can't expose the `@objc` members), forwarding counts/cells to the generic host, which reads straight from the FRC (`controller.sections`/`object(at:)`) — there is no snapshot.

The delegate buffers each cycle's deltas into an `STDataSourceChanges` struct (inserted/deleted/updated/moved items + sections) and `STCollectionViewDataSource.didChangeContent` applies them by the **cheapest correct path**, because the compositional layout's relayout is **O(sections)** and any `performBatchUpdates` — even one that only *reloads* one item — pays that full relayout (~0.1s on a 1557-section/25k library):
- **Pure in-place updates** (the `isSynched`/`isRemote` flip after each uploaded file — by far the most frequent change) reconfigure only the affected **visible** cells in place. **No `performBatchUpdates`, so no relayout**; off-screen rows pick up the new model when scrolled in. This is what removed the per-file "drumbeat" hitch during uploads. Native selection and the checkmark survive (the cell reads its own `isSelected` + the model's `selectedMode`).
- **Structural changes (insert/delete/move) while NOT in selection mode** → a **coalesced, non-animated `reloadData`** (0.15s window via `scheduleCoalescedReload`): a large import's per-file inserts collapse into one relayout instead of one each, and off-window (viewing a photo) UIKit defers it so it can't stutter the viewer. While a coalesced reload is pending the collection view is intentionally out of sync with the FRC, so **every** change folds into that reload (an immediate batch update against stale counts would trap).
- **Structural changes while IN selection mode** (`collectionView.isEditing`) → non-animated `performBatchUpdates` (with item-ops-inside-inserted/deleted-sections filtered out, moves split to delete+insert), because `reloadData` would wipe the collection view's **native** multi-select (the visual checkmarks; the action set `selectionObjectsIdentifiers` is separate and survives either way).

Nothing here is **ever animated**: animating a structural change in a thousands-of-sections layout was the visible ~0.5s hitch; the diff itself is cheap. `STDataBase.DataSource.reloadData()` (the FRC `performFetch`, the O(library) path) now runs **only** on initial load / unlock / sync / a >`maxIncrementalChanges` delta, and stays **synchronous** — the file viewer creates its (flat, `sectionNameKeyPath: nil`) data source and reads a tapped file's index immediately in `viewDidLoad`, so an async fetch there left the FRC un-fetched and trapped (`EXC_BAD_ACCESS`). Every FRC query method also guards on a `hasFetched` flag for the same reason. Batch insert/delete still bypass the contexts; `CollectionProvider.mergeBatchChanges` surfaces their object ids to the `viewContext` via `mergeChanges(fromRemoteContextSave:)` so the FRC emits item-level inserts/deletes (a normal `save()` auto-merges and emits updates directly — see Persistence gotcha 7).

### Tab-bar action accessory (UI gotcha)
The custom `STTabBar` shows an action bar (`STFilesActionTabBarAccessoryView`) for selection mode and the file viewer via its `accessoryView` setter. It is attached as a **sibling on top of the tab bar with the tab bar's `isUserInteractionEnabled` disabled** — *not* as a child (a child gets its taps stolen by the tab bar's nested, re-created buttons). Because it's a sibling, `tabBar.alpha` no longer hides it, so anything that hides the bar (e.g. the viewer's full-screen mode) must fade `accessoryView` explicitly.

### Native camera (`StingleRoot/.../Manager/Camera/` + `Stingle/Scense/Camera/`)
A built-in camera that captures **directly into the encrypted store**, skipping the system Photos app / iCloud. The UI-free **`STCameraEngine`** (StingleRoot) wraps `AVCaptureSession` (all session mutation on a private `sessionQueue`; delegate callbacks marshalled to `responseQueue`) and supports photo, video, slow-mo (high-fps `activeFormat`), time-lapse (`AVCaptureVideoDataOutput` → `STTimeLapseAssembler` streaming to `AVAssetWriter`), and portrait/depth (device-gated via `STCameraDeviceDiscovery`). The stock-style UI (`STCameraVC`/`STCameraVM`, built in code, no storyboard) is added as a **4th gallery tab** in `STGalleryTabBarVC` (programmatically, not in `Gallery.storyboard`).
- **Ingest reuses the share-extension path.** A capture produces an `STCaptureResult` (plaintext temp file + engine-made thumbnail + type/duration); `STCameraImporter.import(result:)` wraps it in `STImporter.CameraFileImportable` and runs `GaleryFileImporter(... uploadIfNeeded: true)`. No new crypto — `createEncryptedFile` is invoked by the importer as usual.
- **Capture works while the app-lock is engaged** (this is the whole point, mirroring Android's separate Camera launcher). `appIsLocked()` only gates the in-memory *private* key; encryption reads the *public* key from disk (`STCrypto.encryptFile` → `Constants.PublicKeyFilename`), and `isFileSystemAvailable`/`canUploadFile()` only need the Core Data user record. So encrypt + background-upload run locked. **Privacy rule:** while `appIsLocked()` the camera shows only the in-session thumbnail and must never browse the library (`didTapThumbnail` is a no-op when locked).
- **Two distinct "locked" states.** (1) *Stingle app-lock* — bypassed in-process: `STCameraLauncher` presents `STCameraVC` full-screen over `STUnlockAppVC` (the previously-empty `didSelectCameraButton` is now wired). (2) *Device passcode-lock* — needs the iOS 18 **`StingleCameraCapture`** `LockedCameraCapture` extension, which reuses `STCameraEngine` + `STCameraImporter` from StingleRoot inside a SwiftUI capture UI.
- **Launch surfaces all converge on `STCameraLauncher`.** URL scheme `stingle://camera` (`SceneDelegate.scene(_:openURLContexts:)` warm, `willConnectTo` cold via `pendingSurface`), home-screen quick action (`UIApplicationShortcutItems` type `org.stingle.photos.shortcut.camera`), Siri App Shortcut (`StingleAppShortcuts`), and the iOS 18 Control Center control (`StingleWidgets` extension). The control/intent (`OpenCameraIntent`, `openAppWhenRun`) can't reference app-only types, so it signals via an **app-group flag** (`STCameraLaunch.setPending()`/`consumePending()`, peek via `isPending`) that `SceneDelegate.sceneDidBecomeActive`, `STMainVC.openAppController`, and `STUnlockAppVC.viewDidAppear` drain. `STCameraVC` (viewWillAppear/viewDidDisappear) toggles `STAppLockUnlocker.Locker.isAutoLockPaused` so a brief backgrounding (permission prompt) can't auto-lock and tear down a session.
  - **Cold-launch timing & the lost-launch bug (each caused a real "camera doesn't open, Face ID shows instead" report).** `handle`/`presentIfPending` must **never present over a not-yet-ready root, and must never drop the request when it can't present yet.** The old code presented as soon as `topMostViewController() != nil`, but on cold launch the root is the transient **`STMainVC` bootstrap router** (it immediately swaps the window root to gallery/lock via `STRootWindowSegue`); presenting over it (or over a not-in-hierarchy controller) failed *and* `sceneDidBecomeActive`'s early `consumePending()` cleared the app-group flag before the real root existed → the camera silently never opened and the locked app fell through to its normal Face ID prompt (this was the quick-action and Control Center bug). Fix: `readyTopMostViewController()` only returns a target once the scene is `foregroundActive`, the root view is in the window hierarchy, and the root is **past `STMainVC`**; otherwise `presentIfPending()` keeps the request pending (the cross-process flag is folded into in-process `pendingSurface` so it isn't lost) for a later drain. It also requires `isLogedIn()` so it never presents the (key-less) camera over the login screen. Drains fire from every safe point: warm `handle`, `sceneDidBecomeActive`, `STMainVC.openAppController` (gallery route only — `goToApp`), and `STUnlockAppVC.viewDidAppear` (the lock route, at the correct post-transition moment).
  - **Camera-over-lock suppresses the unlock screen's auto-biometric prompt (the "User interaction needed" bug).** When the camera launches over `STUnlockAppVC`, that screen's `viewDidAppear → attemptAutoUnlock` would still fire a Face ID/keychain evaluation; with the camera modal on top its `LAContext` sheet can't present → surfaces a "user interaction required" error and races the camera. So `attemptAutoUnlock()` bails while `STCameraLauncher.shared.hasPendingLaunch || .isPresenting`. `viewDidAppear` calls `presentIfPending()` **before** `attemptAutoUnlock()` so the camera is in flight (and `isPresenting` set) by the time the guard runs. Closing the camera re-fires `viewDidAppear` → unlock prompts normally. Don't reintroduce an unconditional auto-prompt here.
- **New targets** (created with the `xcodeproj` ruby gem; objectVersion 52 means every file needs explicit pbxproj registration — there are **no** filesystem-synchronized groups, so new `.swift` files won't compile until added to a target): **`StingleWidgets`** (Widget extension: Control Center control iOS 18 + Lock Screen widget iOS 16) and **`StingleCameraCapture`** (LockedCameraCapture, iOS 18). Both link `StingleRoot`, join the app group, and are embedded in the app's "Embed App Extensions" phase. The `OpenCameraIntent`/`STCameraDeepLink` files are shared (target members of app + widget).
- **Gotchas:** `NSMicrophoneUsageDescription` is **required** (added) or first video capture crashes; `NSLocationWhenInUseUsageDescription` is only used when geotagging is enabled (off by default — no GPS unless opted in; never strip because AVFoundation doesn't embed location unless asked). `LockedCameraCapture` needs the **"Locked Camera Capture" capability** (`com.apple.developer.lockedcamera`) provisioned on the App ID for **device** builds — it builds on the Simulator without it, but the Simulator can't actually run the camera, Control Center controls, lock-screen widgets, or the capture extension, so all of that is **device-only verification**. Settings live in `STAppSettings.camera` (`+Camera.swift`) and deliberately have **no save-to-Photos** option.

## Conventions

- Type prefix `ST` for app types, `STCD` for Core Data managed objects, `I`-prefixed names for protocols (`IRequest`, `IResponse`, `ICDConvertable`).
- Shared, testable, non-UI logic belongs in **StingleRoot** (so the share extension can use it). UI and scenes belong in **Stingle**. The `Stingle/Resources/RootClasses/` folder holds app-target entry points (`AppDelegate`, `SceneDelegate`).
- Note the directory spelling quirks that already exist in the tree: `Scense` (scenes) and `Uitils` (utils). Match existing paths rather than "correcting" them.
- Backend base URL is overridable at runtime via `STApplication.setAppServer(url:)` (stored in the app-group UserDefaults), supporting self-hosted servers.

## Security

This is a security-sensitive, end-to-end-encrypted product. Vulnerabilities are reported privately (see `SECURITY.md`, security@stingle.org). Treat the crypto file format, key management, and the app-group keychain/storage as high-risk surfaces.

### Hardening invariants (learned from the security review — don't regress these)
- **Untrusted header parsing must stay bounds-checked.** `.sp`/`.spk` headers are sealed with `crypto_box_seal`, which is forgeable by anyone who knows the recipient public key — so decrypted header fields are *attacker-controlled*. Every length read from a header is bounded (`Constants.MaxFileHeaderSize`) before allocation, and the inner-header parse in `getFileHeader` uses the local `take(_:)` helper that throws instead of trapping on a short/oversized field. `chunkSize`/`dataSize` are range-validated centrally in `getFileHeader` so the player path (which doesn't re-validate) inherits the guarantee. Same `take` pattern guards `importKeyBundle` and `parseAlbumMetadata`. Don't reintroduce raw `data[offset..<offset+len]` slicing on header bytes.
- **At-rest private-key KDF is `KdfDifficultyLocal` (= Hard/Moderate), not Interactive.** Legacy installs wrapped at `Normal` are upgraded in place transparently by `decryptLocalPrivateKey` (try strong → fall back to legacy → re-wrap). All local wrap/unwrap paths (`generateMainKeypair`, `getPrivateKey`, `reencryptPrivateKey`, `getPrivateKeyFromExportedKey`, `getPrivateKeyForExport`) must go through `KdfDifficultyLocal` / `decryptLocalPrivateKey` — don't hardcode `Normal` for the on-disk key again. Note `getPrivateKey` is now ~0.5–0.7s (Moderate KDF) and runs on a background queue in the biometric unlock path.
- **Backup exclusion + file protection.** The vault lives under the app-group `…/Caches` folder (NOT sandbox `Library/Caches`, so iOS does not auto-exclude it). `STFileSystem.excludeFromBackup(url:)` excludes the vault root and the `Databases` folder. The private key + salt + nonce get `NSFileProtectionComplete` in `savePrivateFile`; **the public key is deliberately left at the default class** because the camera encrypts to it while the device is locked — never raise its protection. The Core Data store uses `completeUntilFirstUserAuthentication` (not `.complete`) because background tasks write it while the device is locked.
- **Biometric secret is Secure-Enclave-gated (device only).** `STKeyChainService.setBiometricSecret/biometricSecret/hasBiometricSecret` store the unlock secret with a `.biometryCurrentSet` `SecAccessControl` (raw Security API — KeychainSwift can't express it). The legacy ungated `passwordKey` is kept as a fallback until the **first confirmed gated read** deletes it, so migration never locks anyone out. On the **Simulator** the gated path is bypassed (`useSecureEnclaveGate` is false — no real Secure Enclave) and the original ungated mechanism is used, so the dev unlock loop is unchanged. `canUnlockApp` is **non-interactive** (must never present a biometric prompt). Because `.biometryCurrentSet` items are invalidated by iOS when the enrolled biometric set changes (Face ID re-enroll, add/remove fingerprint), biometric unlock will then fail with the dedicated `needsPasswordReauth` error (`biometric_needs_password_reauth` string: "unlock once with your password"); a single password unlock self-heals via `storedCredentialMatches` → `setApp`. A stored secret that no longer decrypts the `encryptedPassword` is deleted on the spot so it can't get stuck. This is an accepted trade-off: the security win (forensic/jailbreak can't read the secret) costs a one-time password re-auth after biometric re-enrollment. Device-only verification.
- **ATS:** `NSAllowsArbitraryLoads` was removed; only `NSAllowsLocalNetworking` remains, so a self-hosted server on a public host must be HTTPS (`setAppServer` also rejects non-`https`). Certificate pinning is intentionally NOT implemented (operational coupling with cert rotation + the self-hosted-server feature).
- **Other guards:** `SceneDelegate` adds a blur privacy cover on `sceneWillResignActive` (hides decrypted media from the app-switcher snapshot); the camera auto-lock pause (`isAutoLockPaused`) re-evaluates the lock when cleared and has a 5-min watchdog cap so a stuck flag can't disable auto-lock; `STFileSystem.isSafeFileName` rejects path-traversal in server-supplied file names; `logOut()` wipes `tmp` + local staging; the backup phrase is copied with `.localOnly`/expiry.

### Known residual items (deferred, not yet fixed)
- Session **token is in Core Data**, not the keychain (recoverable from container access independent of app-lock) — moving it touches user persistence + every request builder.
- **In-memory key material is not zeroed** on lock — Swift `[UInt8]` COW + the `STKeyManagement.key` getter handing out buffer copies make a naive `sodium_memzero` misleading; needs a reference-type secure-buffer refactor.
- Per-file `dateCreated`/`dateModified` are sent to the server in cleartext (protocol-level; needs server/Android coordination).
