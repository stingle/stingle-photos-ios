## Overview

Stingle Photos is an end-to-end encrypted photo/video backup app for iOS. Media is encrypted client-side with libsodium before it ever leaves the device; the server only stores ciphertext. Most app logic lives in a shared framework (`StingleRoot`) consumed by both the main app and the share extension.

## Build, Run & Test

There is no Podfile or `Package.swift`; dependencies are resolved via Xcode-managed Swift Package Manager, pinned to released versions (Alamofire 5.12, keychain-swift 24, swift-sodium 0.11 â€” previously tracked `master`). Open `Stingle.xcodeproj` directly â€” there is no `.xcworkspace`.

Targets build with `IPHONEOS_DEPLOYMENT_TARGET = 16.0`, `SWIFT_VERSION = 5.10`, and `SWIFT_STRICT_CONCURRENCY = targeted` (clean). It builds against the current iOS SDK, so behavior follows the latest UIKit/AVFoundation even though the floor is iOS 16.

This repo requires the full Xcode toolchain. `xcodebuild` from the command line needs Xcode selected first (`sudo xcode-select -s /Applications/Xcode.app/Contents/Developer`); the machine may default to CommandLineTools, which cannot build it.

### Deploying to verify (important gotcha)
A command-line `xcodebuild ... -derivedDataPath build` only proves the code **compiles** â€” it writes to `./build` and does **not** update what the user runs from Xcode. The user installs via **Xcode âŒ˜R**, which uses Xcode's *own* DerivedData. So "my build succeeded" â‰  "the app on the device changed." If reported changes don't appear (a recurring trap here): the build is stale â€” **Quit Xcode fully (âŒ˜Q), reopen, Clean Build Folder (â‡§âŒ˜K), âŒ˜R.** Also confirm the running app's logs reflect your edits before chasing a "bug" â€” it's often just a stale build.

To run on a simulator yourself for verification:
```
xcodebuild -project Stingle.xcodeproj -scheme StingleDev -configuration Debug-Dev -destination 'platform=iOS Simulator,name=iPhone 17' -derivedDataPath build build
xcrun simctl install "iPhone 17" build/Build/Products/Debug-Dev-iphonesimulator/Stingle.app
xcrun simctl launch "iPhone 17" org.stingle.photos.dev   # dev bundle id
xcrun simctl io "iPhone 17" screenshot /tmp/shot.png
```
The simulator typically has a saved locked account (`test1@fenritz.com`); real login/sync/upload need credentials to runtime-verify, and the gallery's StoreKit plans need either App Store Connect or the local `Stingle/Resources/Config/Storage.storekit` (already wired into the `StingleDev` scheme).

Targets and schemes:
- **Stingle** (`com.apple.product-type.application`) â€” the main app. Schemes: `StingleDev`, `StingleProd`.
- **StingleShare** (`app-extension`) â€” share-sheet extension for importing media. Schemes: `StingleShareDev`, `StingleShareProd`.
- **StingleRoot** (`framework`) â€” shared core (crypto, DB, network, managers, models, data sources). Scheme: `StingleRoot`. Most non-UI code lives here.

Build configurations are split by environment: `Debug-Dev`, `Debug-Prod`, `Release-Dev`, `Release-Prod`. Dev vs. Prod is driven by xcconfig files in `Stingle/Resources/Config/` (`Development.xcconfig`, `Production.xcconfig`), which set bundle IDs, app name, API URL, and `BUILD_TYPE`. These values are read at runtime through `STEnvironment` (via Info.plist keys like `BUILD_TYPE`, `BASE_API_URL`, `APP_BUNDLE_ID`). Use the `*Dev` schemes for development against the dev backend.

There is no test target in this project; do not assume a `test` action exists.

## Architecture

### App / extension / framework split
- The main app and share extension share an **app group** (`group.<bundleId>.sharing`, see `STEnvironment.groupAppFileSharingBundleId`) for the encrypted file store, UserDefaults, and keychain. Both targets link `StingleRoot`.
- `STEnvironment.current.appIsExtension` distinguishes runtime context (detected by `.appex` bundle suffix). Be careful with code paths that differ between app and extension.

### STApplication â€” the composition root
`STApplication.shared` (in `StingleRoot/.../Services/Application/`) is the singleton that wires everything together and owns the long-lived services: `crypto`, `dataBase`, `syncManager`, `uploader`, `downloaderManager`, `autoImporter`, `appLockUnlocker`, and a per-user `fileSystem`. Logout/delete-account flows that must tear down all state live here (`logoutCashe`/`deleteAccountCashe`). When adding a new global service, register it here.

### Crypto (`Uitils/Crypto/STCrypto*.swift`)
libsodium-based (swift-sodium / Clibsodium). Implements Stingle's custom encrypted file format â€” magic bytes `SP` for media files and `SPK` for key files, chunked XChaCha20-Poly1305 (`__data__` context), versioned headers, and KDF difficulty levels. Keys derive from the user's password + a mnemonic backup phrase. `STCrypto` is split across extensions by concern: `+Header`, `+PrivateKey`, `+Album`, `+Common`, `+Additional`. Format constants (lengths, versions, file-type tags) live in `STCrypto.Constants`. Do not change format constants without understanding on-disk/wire compatibility.

### Persistence (`Manager/DB/`, `Models/DB/`)
Core Data, model name `StingleModel` (`.xcdatamodeld` in `StingleRoot/Resources/DBModel/`, with a migration mapping model present). `STDataBase` exposes typed **providers** â€” `userProvider`, `dbInfoProvider`, and `SyncProvider`s for gallery, albums, albumFiles, trash, and contacts. Each library entity has a Core Data class (`STCD*`) and a plain model (`STLibrary.*`) bridged via `CDConvertable`/`ICDSynchConvertable`. Sync diffs are modeled as `SyncInfo` sets (inserts/updates/upgrade/deletes).

### Sync
`STSyncManager` reconciles local Core Data with the server. The server is authoritative for the encrypted catalog; the client pulls deltas keyed off `STDBInfo` timestamps and applies them through the sync providers. Sync runs on unlock and is gated by `appLockUnlocker`.

### Network (`Network/`)
Layered: `Request/` (typed `IRequest` definitions per domain â€” Auth, Album, File, Sync, Upload, etc.) â†’ `Dispatcher/` (`STNetworkDispatcher` + `STNetworkSession`, Alamofire-backed, with request encoding) â†’ `Worker/` (high-level domain operations like `STSyncWorker`, `STUploadWorker`, `STContactWorker`) â†’ `Response/` (`IResponse`/`STResponse`). `Operations/` (`STOperationManager` + queues) drives concurrent background work for uploads/downloads.

### Concurrency / async-await (cross-cutting)
The codebase was historically all completion-handler callbacks. The worker layer now also exposes `async throws` siblings (in `STWorker` and each worker) that **bridge the existing callback methods via `withCheckedThrowingContinuation`** â€” both styles coexist so migration is incremental. View models call the async API inside `Task { @MainActor [weak self] in ... }`, which preserves the original "callbacks fire on main" semantics (the operation layer doesn't hop threads; Alamofire's default queue is `.main`).

**Continuation hazard:** a bridged callback method *must* invoke success XOR failure exactly once. A path that fires both â‡’ `continuation.resume` twice â‡’ **crash**; a path that fires neither â‡’ leaked continuation/hang. The operation layer nils its closures in `setFinished` (so post-success cancel is safe), and `STOperation.setCancel` fires a `.cancelled` failure on ready-cancel so awaiters don't leak. When you bridge or edit a callback worker method, verify every branch resolves exactly once.

### File transfer & storage
- `Manager/FileUploader/` â€” `STFileUploader` + per-file `STFileUploaderOperation`; encrypts and uploads.
  - **Uploading is gated separately from auto-import.** `canUploadFile()` requires **backup** enabled (`STAppSettings.backup.isEnabled`, *not* the auto-import toggle), Wi-Fi if Wi-Fi-only, and battery â‰¥ threshold. The battery check uses `UIDevice.current.batteryLevel`, which is **-1 unless `isBatteryMonitoringEnabled`** â€” enabled in `AppDelegate` at launch, and `canUploadFile()` ignores an unknown (-1) level so a cold background launch isn't blocked.
  - **Background uploads.** File upload requests are always `isBackground = true` and run on the `URLSessionConfiguration.background` session in `STNetworkSession` (`sessionSendsLaunchEvents = true`, app-group `sharedContainerIdentifier`), so byte transfers continue after suspension. `AppDelegate.application(_:handleEventsForBackgroundURLSession:â€¦)` forwards the system completion handler via `STNetworkDispatcher.handleEventsForBackgroundURLSession(...)` so completions get finalized when iOS wakes the app â€” without this, transfers finish but the localâ†’remote DB finalization waits for the next foreground launch. Note the BG auto-import task signals completion when *encryption/import* finishes, not when uploads finish; uploads then ride the background session independently.
- `Manager/FileDownloader/` â€” `STDownloaderManager` with a `Cache/` layer for decrypted thumbnails/originals.
- `Manager/FileSystem/STFileSystem` â€” per-user on-disk layout under the user's `homeFolder`; handles migration, logout cleanup, free-disk checks (`STConstants.minFreeDiskUnits`).
- `Importer/` â€” `STImporter` and `STFileAutoImporter` (`AutoImporter`) ingest from the photo library / share extension, including background auto-import.
  - **Auto-import triggering (reliability gotchas, learned the hard way).** `AutoImporter.startImport()` is the single entry point and self-guards via `canStartImport` (logged-in + auto-import enabled + filesystem available). It's now driven by four triggers so it doesn't stall when the app is closed or idle: (1) a `PHPhotoLibraryChangeObserver` on `AutoImporter` (so `AutoImporter` is an `NSObject`) imports new media live; (2) `STSyncManager.didEndSync` calls it after *every* sync attempt â€” success **and** failure (don't move it back into the sync success closure, or a failed/offline sync skips import); (3) two `BGTaskScheduler` tasks in `STBGTaskScheduler` â€” a `BGProcessingTaskRequest` (`org.stingle.photos.auto.import`) and a `BGAppRefreshTaskRequest` (`â€¦import.refresh`); both ids must stay in the app `Info.plist` `BGTaskSchedulerPermittedIdentifiers` and `UIBackgroundModes` must include both `processing` and `fetch`. Background processing no longer sets `requiresExternalPower` (it used to, so imports only ran while charging). **`STBGTaskScheduler.shared.start()` (which calls `BGTaskScheduler.register`) must run synchronously inside `AppDelegate.didFinishLaunchingWithOptions` â€” it was previously wrapped in `DispatchQueue.main.async`, which registers the handlers one run-loop tick too late, and iOS then silently never launches the app to run them (the #1 reason "background auto-import doesn't work at all").** Note iOS also won't run BG tasks after the user force-quits the app, and BGTaskScheduler never fires on the Simulator (test on a real device, or trigger via the LLDB `_simulateLaunchForTaskWithIdentifier:` private call) â€” foreground/observer triggers are the real safety net. Background import doesn't need an app unlock: `canStartImport` is Core-Data-backed and new files seal to the user's public key. (4) the settings toggle.
  - `startImport()` coalesces re-triggers during an in-flight run via `needsRestart` (re-runs once on completion) so a change/sync arriving mid-import isn't dropped.
  - **Detection is creationDate-paged + identifier-deduped.** The scan fetches `creationDate >= lastImportDate` ascending and skips assets whose `localIdentifier` is in a persisted, bounded (`importedIdentifiersCap`) set (`UserDefaults` key `autoImporter.importedIdentifiers`). The `>=` (was `>`) plus dedup fixes assets that share the boundary timestamp being silently skipped, without re-importing or looping. `resetImportDate` clears the id set (logout / "reset import date" / enabling auto-import, which baselines `lastImportDate` to *now*). Known limitation: media saved *now* but with a `creationDate` older than the baseline (Messages/AirDrop/iCloud) is intentionally not imported â€” "import from enable point" semantics.

### UI (`Stingle/`)
- UIKit, organized by feature under `Stingle/Scense/` (Gallery, Home, Welcome, Settings, Shear/share, ImageEditor). Storyboards + xibs are used alongside code.
- Screens follow an **MVVM** convention: `ST<Feature>VC` (view controller) paired with `ST<Feature>VM` (view model), e.g. `STGalleryVC` / `STGalleryVM`.
- Collection-based screens bind to Core Data through `STViewDataSource` / `STCollectionViewDataSource` (in `StingleRoot/.../DataSource/`), which wrap `NSDiffableDataSource` snapshots fed by the DB providers. Reusable views/controls live in `Stingle/Common/UI/`.

### App lock
`Manager/AppLockUnlocker/` + `STBiometricAuthServices` implement passcode/biometric locking. Locking pauses sync and clears in-memory key material; unlocking re-triggers sync and DB reload (`STApplication.appDidLocked`). The biometric credential is an encrypted password in app-group UserDefaults with its key in the keychain (`STKeyChainService`). Two non-obvious rules learned here: (1) that keychain item must use `accessibleAfterFirstUnlockThisDeviceOnly` (not the `WhenUnlocked` default) or Face ID fails after a long device lock and falls back to the password dialog; (2) decide whether to *offer* biometrics from `isBiometricConfigured` (UserDefaults-backed, reliable) rather than a read-and-validate that can fail transiently, and defer the auto-attempt until `protectedDataDidBecomeAvailable`.

### Video playback (`Uitils/Player/`)
`STPlayer` wraps `AVPlayer`; `STAssetResourceLoader` is an `AVAssetResourceLoaderDelegate` that decrypts the chunked stream on the fly (local file or network range-request reader). Each chunk decrypts independently (per-chunk key derived from the chunk number), so the asset **does** support random access â€” `contentInformationRequest.isByteRangeAccessSupported` must be `true` for scrubbing to work at all. Seeking is coalesced ("chase time") with a tolerance so a drag doesn't flood `AVPlayer`. Video viewers (`Items/Video/STVideoViewerVC`) autoplay when they become the current page (`fileViewer(activateContent:)` from `STFileViewerVC.didChangeFileViewer`) and pause off-screen.

### Tab-bar action accessory (UI gotcha)
The custom `STTabBar` shows an action bar (`STFilesActionTabBarAccessoryView`) for selection mode and the file viewer via its `accessoryView` setter. It is attached as a **sibling on top of the tab bar with the tab bar's `isUserInteractionEnabled` disabled** â€” *not* as a child (a child gets its taps stolen by the tab bar's nested, re-created buttons). Because it's a sibling, `tabBar.alpha` no longer hides it, so anything that hides the bar (e.g. the viewer's full-screen mode) must fade `accessoryView` explicitly.

### Native camera (`StingleRoot/.../Manager/Camera/` + `Stingle/Scense/Camera/`)
A built-in camera that captures **directly into the encrypted store**, skipping the system Photos app / iCloud. The UI-free **`STCameraEngine`** (StingleRoot) wraps `AVCaptureSession` (all session mutation on a private `sessionQueue`; delegate callbacks marshalled to `responseQueue`) and supports photo, video, slow-mo (high-fps `activeFormat`), time-lapse (`AVCaptureVideoDataOutput` â†’ `STTimeLapseAssembler` streaming to `AVAssetWriter`), and portrait/depth (device-gated via `STCameraDeviceDiscovery`). The stock-style UI (`STCameraVC`/`STCameraVM`, built in code, no storyboard) is added as a **4th gallery tab** in `STGalleryTabBarVC` (programmatically, not in `Gallery.storyboard`).
- **Ingest reuses the share-extension path.** A capture produces an `STCaptureResult` (plaintext temp file + engine-made thumbnail + type/duration); `STCameraImporter.import(result:)` wraps it in `STImporter.CameraFileImportable` and runs `GaleryFileImporter(... uploadIfNeeded: true)`. No new crypto â€” `createEncryptedFile` is invoked by the importer as usual.
- **Capture works while the app-lock is engaged** (this is the whole point, mirroring Android's separate Camera launcher). `appIsLocked()` only gates the in-memory *private* key; encryption reads the *public* key from disk (`STCrypto.encryptFile` â†’ `Constants.PublicKeyFilename`), and `isFileSystemAvailable`/`canUploadFile()` only need the Core Data user record. So encrypt + background-upload run locked. **Privacy rule:** while `appIsLocked()` the camera shows only the in-session thumbnail and must never browse the library (`didTapThumbnail` is a no-op when locked).
- **Two distinct "locked" states.** (1) *Stingle app-lock* â€” bypassed in-process: `STCameraLauncher` presents `STCameraVC` full-screen over `STUnlockAppVC` (the previously-empty `didSelectCameraButton` is now wired). (2) *Device passcode-lock* â€” needs the iOS 18 **`StingleCameraCapture`** `LockedCameraCapture` extension, which reuses `STCameraEngine` + `STCameraImporter` from StingleRoot inside a SwiftUI capture UI.
- **Launch surfaces all converge on `STCameraLauncher`.** URL scheme `stingle://camera` (`SceneDelegate.scene(_:openURLContexts:)` warm, `willConnectTo` cold via `pendingSurface`), home-screen quick action (`UIApplicationShortcutItems` type `org.stingle.photos.shortcut.camera`), Siri App Shortcut (`StingleAppShortcuts`), and the iOS 18 Control Center control (`StingleWidgets` extension). The control/intent (`OpenCameraIntent`, `openAppWhenRun`) can't reference app-only types, so it signals via an **app-group flag** (`STCameraLaunch.setPending()`/`consumePending()`) that `SceneDelegate.sceneDidBecomeActive` and `STMainVC.openAppController` drain. `STCameraVC` (viewWillAppear/viewDidDisappear) toggles `STAppLockUnlocker.Locker.isAutoLockPaused` so a brief backgrounding (permission prompt) can't auto-lock and tear down a session.
- **New targets** (created with the `xcodeproj` ruby gem; objectVersion 52 means every file needs explicit pbxproj registration â€” there are **no** filesystem-synchronized groups, so new `.swift` files won't compile until added to a target): **`StingleWidgets`** (Widget extension: Control Center control iOS 18 + Lock Screen widget iOS 16) and **`StingleCameraCapture`** (LockedCameraCapture, iOS 18). Both link `StingleRoot`, join the app group, and are embedded in the app's "Embed App Extensions" phase. The `OpenCameraIntent`/`STCameraDeepLink` files are shared (target members of app + widget).
- **Gotchas:** `NSMicrophoneUsageDescription` is **required** (added) or first video capture crashes; `NSLocationWhenInUseUsageDescription` is only used when geotagging is enabled (off by default â€” no GPS unless opted in; never strip because AVFoundation doesn't embed location unless asked). `LockedCameraCapture` needs the **"Locked Camera Capture" capability** (`com.apple.developer.lockedcamera`) provisioned on the App ID for **device** builds â€” it builds on the Simulator without it, but the Simulator can't actually run the camera, Control Center controls, lock-screen widgets, or the capture extension, so all of that is **device-only verification**. Settings live in `STAppSettings.camera` (`+Camera.swift`) and deliberately have **no save-to-Photos** option.

## Conventions

- Type prefix `ST` for app types, `STCD` for Core Data managed objects, `I`-prefixed names for protocols (`IRequest`, `IResponse`, `ICDConvertable`).
- Shared, testable, non-UI logic belongs in **StingleRoot** (so the share extension can use it). UI and scenes belong in **Stingle**. The `Stingle/Resources/RootClasses/` folder holds app-target entry points (`AppDelegate`, `SceneDelegate`).
- Note the directory spelling quirks that already exist in the tree: `Scense` (scenes) and `Uitils` (utils). Match existing paths rather than "correcting" them.
- Backend base URL is overridable at runtime via `STApplication.setAppServer(url:)` (stored in the app-group UserDefaults), supporting self-hosted servers.

## Security

This is a security-sensitive, end-to-end-encrypted product. Vulnerabilities are reported privately (see `SECURITY.md`, security@stingle.org). Treat the crypto file format, key management, and the app-group keychain/storage as high-risk surfaces.

### Hardening invariants (learned from the security review â€” don't regress these)
- **Untrusted header parsing must stay bounds-checked.** `.sp`/`.spk` headers are sealed with `crypto_box_seal`, which is forgeable by anyone who knows the recipient public key â€” so decrypted header fields are *attacker-controlled*. Every length read from a header is bounded (`Constants.MaxFileHeaderSize`) before allocation, and the inner-header parse in `getFileHeader` uses the local `take(_:)` helper that throws instead of trapping on a short/oversized field. `chunkSize`/`dataSize` are range-validated centrally in `getFileHeader` so the player path (which doesn't re-validate) inherits the guarantee. Same `take` pattern guards `importKeyBundle` and `parseAlbumMetadata`. Don't reintroduce raw `data[offset..<offset+len]` slicing on header bytes.
- **At-rest private-key KDF is `KdfDifficultyLocal` (= Hard/Moderate), not Interactive.** Legacy installs wrapped at `Normal` are upgraded in place transparently by `decryptLocalPrivateKey` (try strong â†’ fall back to legacy â†’ re-wrap). All local wrap/unwrap paths (`generateMainKeypair`, `getPrivateKey`, `reencryptPrivateKey`, `getPrivateKeyFromExportedKey`, `getPrivateKeyForExport`) must go through `KdfDifficultyLocal` / `decryptLocalPrivateKey` â€” don't hardcode `Normal` for the on-disk key again. Note `getPrivateKey` is now ~0.5â€“0.7s (Moderate KDF) and runs on a background queue in the biometric unlock path.
- **Backup exclusion + file protection.** The vault lives under the app-group `â€¦/Caches` folder (NOT sandbox `Library/Caches`, so iOS does not auto-exclude it). `STFileSystem.excludeFromBackup(url:)` excludes the vault root and the `Databases` folder. The private key + salt + nonce get `NSFileProtectionComplete` in `savePrivateFile`; **the public key is deliberately left at the default class** because the camera encrypts to it while the device is locked â€” never raise its protection. The Core Data store uses `completeUntilFirstUserAuthentication` (not `.complete`) because background tasks write it while the device is locked.
- **Biometric secret is Secure-Enclave-gated (device only).** `STKeyChainService.setBiometricSecret/biometricSecret/hasBiometricSecret` store the unlock secret with a `.biometryCurrentSet` `SecAccessControl` (raw Security API â€” KeychainSwift can't express it). The legacy ungated `passwordKey` is kept as a fallback until the **first confirmed gated read** deletes it, so migration never locks anyone out. On the **Simulator** the gated path is bypassed (`useSecureEnclaveGate` is false â€” no real Secure Enclave) and the original ungated mechanism is used, so the dev unlock loop is unchanged. `canUnlockApp` is **non-interactive** (must never present a biometric prompt). Because `.biometryCurrentSet` items are invalidated by iOS when the enrolled biometric set changes (Face ID re-enroll, add/remove fingerprint), biometric unlock will then fail with the dedicated `needsPasswordReauth` error (`biometric_needs_password_reauth` string: "unlock once with your password"); a single password unlock self-heals via `storedCredentialMatches` â†’ `setApp`. A stored secret that no longer decrypts the `encryptedPassword` is deleted on the spot so it can't get stuck. This is an accepted trade-off: the security win (forensic/jailbreak can't read the secret) costs a one-time password re-auth after biometric re-enrollment. Device-only verification.
- **ATS:** `NSAllowsArbitraryLoads` was removed; only `NSAllowsLocalNetworking` remains, so a self-hosted server on a public host must be HTTPS (`setAppServer` also rejects non-`https`). Certificate pinning is intentionally NOT implemented (operational coupling with cert rotation + the self-hosted-server feature).
- **Other guards:** `SceneDelegate` adds a blur privacy cover on `sceneWillResignActive` (hides decrypted media from the app-switcher snapshot); the camera auto-lock pause (`isAutoLockPaused`) re-evaluates the lock when cleared and has a 5-min watchdog cap so a stuck flag can't disable auto-lock; `STFileSystem.isSafeFileName` rejects path-traversal in server-supplied file names; `logOut()` wipes `tmp` + local staging; the backup phrase is copied with `.localOnly`/expiry.

### Known residual items (deferred, not yet fixed)
- Session **token is in Core Data**, not the keychain (recoverable from container access independent of app-lock) â€” moving it touches user persistence + every request builder.
- **In-memory key material is not zeroed** on lock â€” Swift `[UInt8]` COW + the `STKeyManagement.key` getter handing out buffer copies make a naive `sodium_memzero` misleading; needs a reference-type secure-buffer refactor.
- Per-file `dateCreated`/`dateModified` are sent to the server in cleartext (protocol-level; needs server/Android coordination).
